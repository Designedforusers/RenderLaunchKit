# LaunchKit: A Multi-Agent Pipeline on Render

Paste a GitHub URL. Get back a complete go-to-market kit — blog posts, product images, demo videos with Remotion compositing, voiceovers, podcast scripts, 3D world scenes — generated by AI agents that research your product before creating anything.

One TypeScript monorepo. Eight Render services. Here's how it works under the hood.

## The pipeline

Six stages, each backed by a different agent or processor:

1. **Analyze** the repo — README, file tree, dependencies, framework, license.
2. **Research** the market — autonomous web searches via the Claude Agent SDK. Competitors, Hacker News discussions, developer sentiment.
3. **Strategize** a launch plan — opinionated, tailored to the specific product, with per-asset generation instructions.
4. **Generate** the full kit — fanned out in parallel via Render Workflows, with per-task compute sizing (Starter for text, Standard for images, Pro for video and 3D).
5. **Review** everything — a creative-director agent scores each asset, auto-approves the good ones, re-queues rejects with revision instructions.
6. **Learn** from feedback — user edits feed a self-learning loop. Voyage embeddings, pgvector similarity clustering, Claude Haiku compression. The next project benefits from every correction.

Every dollar spent on AI providers is tracked per-asset via `AsyncLocalStorage`. The dashboard shows real costs, not estimates. When it says "Generated for $0.11," that's the actual Anthropic + fal.ai + ElevenLabs spend.

## Why eight services

Each service exists for a specific architectural reason. The project didn't start with eight — it started with three and added five when the deployment constraints demanded it.

**Render Workflows** is the centerpiece. Text generation runs on a Starter task (a Claude API call, done in 3 seconds). Video rendering runs on Pro (headless Chrome, 2 CPU, 90 seconds). One pipeline, five compute profiles. Without per-task compute isolation, you'd either overprovision for text or underprovision for video.

**Two Docker services** handle system-level dependencies the Node runtime can't install. The Remotion renderer needs Chrome's system libraries for video compositing. The Pika worker needs Python for a vendored subprocess. Both run `node:22-slim` with `apt-get` in the Dockerfile.

**BullMQ on Key-Value (Redis)** does triple duty: job queues for the analyze→research→strategize→review pipeline, pub/sub for SSE progress events to the dashboard, and a TTL cache for GitHub API responses. Three patterns, one managed instance, `noeviction` policy so jobs never get silently dropped.

**Postgres** stores relational state and pgvector embeddings. A dedicated vector database wasn't worth the operational overhead — at this scale (hundreds of embeddings, not millions), a `pgvector` index on existing Postgres is simpler to operate and one fewer service to manage.

**MinIO on a Render Disk** provides S3-compatible object storage without leaving the platform. Rendered MP4s upload here; the web service 302-redirects clients to the public MinIO URL instead of streaming bytes through the API.

**Cron** runs every six hours: aggregate user feedback into embedding clusters, refresh trending developer topics from seven sources (Grok live X search, Hacker News, dev.to, Reddit, Product Hunt, GitHub, Exa).

| Service | Type | Runtime | Role |
|---|---|---|---|
| `launchkit-web` | Web | Node | Hono API + React SPA + SSE + webhooks |
| `launchkit-worker` | Worker | Node | BullMQ agent loops (30s–5min each) |
| `launchkit-pika-worker` | Worker | Docker | Isolated Python subprocess for video meetings |
| `launchkit-cron` | Cron | Node | 6-hourly trending + feedback clustering |
| `launchkit-workflows` | Workflow | Node | Per-asset generation with per-task sizing |
| `launchkit-renderer` | Web | Docker | Remotion MP4 compositing with Chrome |
| `launchkit-minio` | Web | Docker image | S3-compatible storage on a Render Disk |
| `launchkit-db` | Postgres | — | Relational state + pgvector |
| `launchkit-redis` | Key-Value | — | Queues + pub/sub + cache |

## The Remotion renderer split

This is the deployment decision worth walking through.

Render Workflows is in public beta. The build environment has a read-only filesystem — `apt-get` fails. Remotion needs Chrome's system libraries (`libnss3`, `libatk1.0-0`, `libgbm-dev`, and about ten others). Six of seven generation tasks are pure API calls that don't need system packages. One task needs a browser.

The fix: keep Workflows for what it's great at (orchestration + compute-sized API tasks), add a Docker-based web service for the one browser-native workload. Same `createRemotionRenderer` code, same MinIO upload, just HTTP transport instead of the Workflows task protocol. The web service's `triggerRemotionRender` helper changed from calling `render.workflows.startTask()` to calling `POST /render` on the renderer service. One file changed, same typed interface.

The split actually made the architecture cleaner — it's a real service boundary, not a workaround. When Workflows exits beta and supports Docker runtimes, the boundary is clean enough to reverse.

## Engineering discipline

- **TypeScript strict mode + four extra flags** — `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `noPropertyAccessFromIndexSignature`, `noImplicitOverride`. Zero `@ts-ignore`. Two `as unknown as` casts in the entire codebase, both centralized with docstrings.
- **Zod at every runtime boundary** — HTTP bodies, LLM responses, env vars, `jsonb` columns, external APIs, webhooks. No raw input touches application code.
- **ESLint at `--max-warnings=0`** with the anti-`any` family enabled.
- **170 passing tests** covering schemas, pricing helpers, auth middleware, crypto, and API response shapes.
- **`AsyncLocalStorage` cost tracking** — every provider call records actual spend without polluting function signatures. Non-blocking: cost tracking failures never block asset generation (four layers of protection).
- **Lefthook pre-push** runs typecheck → lint → build before every push. CI mirrors the same chain.

## Patterns worth stealing

If you're building something similar on Render:

- **Per-task compute isolation on Render Workflows.** Starter for text, Standard for images/audio, Pro for video and 3D. The parent task fans out with `Promise.allSettled` and handles partial failures as first-class outcomes. (`apps/workflows/src/tasks/`)
- **`AsyncLocalStorage` for cross-cutting concerns** without parameter pollution. Used here for cost tracking; works equally well for request tracing, audit logging, or tenant isolation. (`docs/cost-tracking.md`)
- **pgvector on your existing Postgres** instead of a dedicated vector database. At this scale, the operational simplicity wins. (`packages/shared/src/schemas/`)
- **Strict TypeScript with four extra flags.** It slows you down for the first day. After that, the compiler catches real bugs that tests wouldn't. (`tsconfig.json`)
- **Self-learning via embedding clusters.** User edits → Voyage embeddings → pgvector similarity → Claude Haiku summarization → imperative directives → prompt context for the next generation. The loop closes automatically via cron. (`apps/worker/src/processors/`, `apps/cron/src/`)

## Known limitations

Three things that would change with more time:

1. **SSE stream is unauthenticated.** The browser's `EventSource` API doesn't support custom headers. The proper fix is a signed HMAC token in the query string. Rate limiting and UUID opacity provide baseline protection today.
2. **No pagination on the project list.** `GET /api/projects` returns everything. Fine for demo, not fine for production.
3. **The renderer is a separate service because of a beta constraint.** When Render Workflows supports Docker runtimes, the renderer can collapse back into a single Workflows task. The service boundary is clean enough to reverse.
