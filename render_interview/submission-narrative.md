# LaunchKit: What Happens When You Point a Multi-Agent Pipeline at Render

I built LaunchKit to answer a question I couldn't stop thinking about: what if you could paste a GitHub URL and get back a complete go-to-market kit — blog posts, product images, demo videos, voiceovers, podcast scripts, 3D scenes — generated by AI agents that actually research your product before creating anything?

The answer, it turns out, is a TypeScript monorepo with eight Render services, 200+ commits, and a deployment story worth telling.

## What LaunchKit does

The pipeline has six stages, each backed by a different agent or processor:

1. **Analyze** the repo — README, file tree, dependencies, framework, license.
2. **Research** the market — autonomous web searches via the Claude Agent SDK. Competitors, Hacker News discussions, developer sentiment.
3. **Strategize** a launch plan — opinionated, tailored to the specific product, with per-asset generation instructions.
4. **Generate** the full kit — fanned out in parallel via Render Workflows, with per-task compute sizing.
5. **Review** everything — a creative-director agent scores each asset, auto-approves high quality, re-queues rejects.
6. **Learn** from feedback — user edits feed a self-learning loop via pgvector similarity clustering. The next project benefits from every correction.

Every dollar spent on AI providers is tracked per-asset via `AsyncLocalStorage`. The dashboard shows real costs, not estimates. When it says "Generated for $0.11," that's the actual Anthropic + fal.ai + ElevenLabs spend, broken down by asset.

## Why eight services

Each service exists for a specific architectural reason. I didn't start with eight — I started with three and added five when the deployment constraints demanded it.

**Render Workflows** is the centerpiece. Text generation runs on a Starter task (a Claude API call, done in 3 seconds). Video rendering runs on Pro (headless Chrome, 2 CPU, 90 seconds). One pipeline, five compute profiles. This is the feature that made the architecture possible — without per-task compute isolation, I'd either overprovision for text or underprovision for video.

**Two Docker services** handle system-level dependencies the Node runtime can't install. The Remotion renderer needs Chrome's system libraries for video compositing. The Pika worker needs Python for a vendored subprocess. Both run `node:22-slim` with `apt-get` in the Dockerfile.

**BullMQ on Key-Value (Redis)** does triple duty: job queues for the analyze→research→strategize→review pipeline, pub/sub for SSE progress events to the dashboard, and a TTL cache for GitHub API responses. Three patterns, one managed instance.

**Postgres** stores relational state and pgvector embeddings for the self-learning loop. I considered a dedicated vector database and decided against it — at this scale, a `pgvector` index on an existing Postgres instance is simpler to operate and one fewer service to manage.

**MinIO on a Render Disk** provides S3-compatible object storage without leaving the platform. Rendered MP4s upload here; the web service 302-redirects clients to the public MinIO URL instead of streaming bytes through the API instance.

**Cron** runs every six hours: aggregate user feedback into embedding clusters, refresh trending developer topics from GitHub, Hacker News, Product Hunt, and X.

The full service inventory:

| `launchkit-web` | Web | Node | Pro | Hono API + React SPA + SSE + webhooks |

## Seven things I learned deploying it

Here's the thing about this project: 170 tests passed locally. Zero application-code bugs in production. Every single deploy failure was an infrastructure configuration issue — not a logic error, not a type error, not a missing dependency. The platform was enforcing production constraints that my local environment didn't surface.

**1. `NODE_ENV=production` strips devDependencies — but TypeScript type packages live there.**

`npm ci` in production mode skips `@types/react`, `@types/node`, and every other dev dependency. The build fails with missing type declarations. Fix: `npm ci --include=dev && npm run build`. Two flags, five deploy attempts to figure out.

**2. Postgres plan names changed.**

I wrote `plan: starter` in my first `render.yaml`. Then `plan: pro`. Then `plan: pro_4gb`. The actual value is `pro-4gb` — hyphenated, with the size suffix. Three commits to get a string right. You can see them in the git log:

```
6c80d54 fix(deploy): update Postgres plan from legacy starter to basic_256mb
6567cc4 fix(deploy): use 'pro' plan for Postgres (not pro_4gb)
2648332 fix(deploy): correct Postgres plan name to pro-4gb (hyphenated)
```

**3. Render Workflows has a read-only filesystem.**

Workflows is in public beta, and `apt-get` fails with `E: List directory /var/lib/apt/lists/partial is missing. - Acquire (30: Read-only file system)`. This matters because Remotion needs Chrome's system libraries — `libnss3`, `libatk1.0-0`, `libgbm-dev`, and about ten others.

I solved this by splitting the Remotion render task into a dedicated Docker service (`launchkit-renderer`) and calling it via HTTP from Workflows. Same `createRemotionRenderer` code, same MinIO upload, just HTTP transport instead of the Workflows task protocol. The six generation tasks that are pure API calls stayed on Workflows where they belong.

This is the deployment decision I'd walk through in any architecture conversation — it's a pragmatic workaround with a clean service boundary, not a hack.

**4. Docker builds without `.dockerignore` are a trap.**

The first Docker deploy copied 1GB+ of macOS-built `node_modules` into the build context, shadowing the clean Linux `npm ci` install. Native binaries didn't resolve. Three lines fixed it:

```
node_modules
.git
dist
```

**5. Husky and Lefthook `prepare` scripts need `git` in the Docker image.**

If `package.json` has a `prepare` script (and mine does — Lefthook installs its hooks there), `npm ci` runs it and fails with `git: not found`. One `apt-get install -y git` in the Dockerfile.

**6. Remotion's Chrome download is unreliable at build time in monorepos.**

`npx remotion browser ensure` during a Docker build downloads Chrome Headless Shell — but in a monorepo with workspaces, the download path can land in the wrong `node_modules`. Remotion's own docs recommend installing system libraries at build time and letting `ensureBrowser()` handle Chrome at runtime. I burned four commits learning this:

```
85f0dc3 fix(deploy): reorder Dockerfile steps — full monorepo needed before npx remotion
de2d588 fix(deploy): skip npx remotion browser ensure in Docker build — runtime download
94684ee fix(deploy): add .dockerignore and match Remotion's official Dockerfile
3553b3b fix(deploy): remove npx remotion browser ensure from Dockerfile
```

**7. Workspace policies can add Blueprint requirements you don't expect.**

My `render.yaml` Blueprint was valid according to the spec, but the workspace policy required an explicit `ipAllowList` on Key-Value services. The error doesn't mention workspace policies — it just says the allow list is required. Two commits:

```
d03f2d2 fix(deploy): add ipAllowList to Redis Key-Value service
7ca7960 fix(deploy): use explicit allow-all IP list for Redis Key-Value
```

## Engineering discipline

I treated this codebase the way I'd treat production code at a job I cared about:

- **Zero TypeScript escape hatches.** `strict: true` plus four extra flags — `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `noPropertyAccessFromIndexSignature`, `noImplicitOverride`. Zero `@ts-ignore`. Zero `@ts-expect-error`. Two `as unknown as` casts in the entire codebase, both centralized with docstrings explaining the type system limitation they bridge.
- **Zod at every runtime boundary.** HTTP request bodies, LLM responses, environment variables, `jsonb` columns, external API payloads, GitHub webhooks — all validated through Zod schemas before the application code touches them.
- **ESLint at `--max-warnings=0`** with the anti-`any` family enabled (`no-explicit-any`, `no-unsafe-assignment`, `no-unsafe-member-access`, `no-unsafe-call`, `no-unsafe-return`).
- **170 passing tests** covering schemas, pricing helpers, auth middleware, crypto, trigger interfaces, and API response shapes.
- **Real cost tracking** via `AsyncLocalStorage` — every Anthropic, fal.ai, and ElevenLabs call records its actual cost without polluting function signatures. Non-blocking invariant: cost tracking failures never block asset generation (four layers of protection).

The `render.yaml` has inline comments explaining what every environment variable does and which services share it. The commit history follows conventional commits. The `README.md` has a "Patterns you can steal" section with specific file paths.

## What I'd change

Three things, if I had more time:

1. **Authenticate the SSE stream.** `/api/projects/:id/events` is open. The `EventSource` API doesn't support custom headers, so the proper fix is a signed HMAC token in the query string. Rate limiting and UUID opacity provide baseline protection today.
2. **Add cursor-based pagination** to the project list endpoint. `GET /api/projects` returns everything. Fine for demo scale, not fine for production.
3. **Collapse the renderer back into Workflows** when Workflows exits beta and (presumably) supports Docker runtimes. The HTTP sidecar works, but it's unnecessary indirection once the filesystem constraint is lifted.

## Patterns you can steal

If you're building something similar on Render, here's what I think is worth lifting:

- **Per-task compute isolation on Render Workflows.** Starter for text, Standard for images/audio, Pro for video and 3D. The parent task fans out with `Promise.allSettled` and handles partial failures as first-class outcomes.
- **`AsyncLocalStorage` for cross-cutting concerns** without parameter pollution. I use it for cost tracking; you could use it for request tracing, audit logging, or tenant isolation.
- **pgvector on your existing Postgres** instead of a dedicated vector database. At my scale (hundreds of embeddings, not millions), the operational simplicity wins.
- **Strict TypeScript with four extra flags and zero escape hatches.** It slows you down for the first day. After that, the compiler catches real bugs that tests wouldn't.
- **Self-learning via embedding clusters.** User edits → Voyage embeddings → pgvector similarity clustering → Claude Haiku summarization → imperative directives → prompt context for the next generation. The loop closes automatically via cron.
