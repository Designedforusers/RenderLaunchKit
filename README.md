# LaunchKit — AI Go-To-Market Teammate

> Paste a GitHub repo. Get a launch strategy and marketing kit generated by an AI team that researches, strategizes, creates, and learns from every project.

[Live Demo](https://launchkit.onrender.com) · [Deploy Your Own](#deploy-your-own) · [Architecture](#architecture--and-why)

---

## What It Does

LaunchKit is an AI-powered go-to-market teammate. You give it a GitHub repo URL. It does the rest:

1. **Analyzes** the repo — README, file tree, dependencies, recent commits, framework, language
2. **Researches** the market autonomously — Claude uses tool calls to search GitHub, the web, and Hacker News for competitors and discussions
3. **Strategizes** an opinionated launch plan — picks the right channels, tone, and asset types for *this specific* product
4. **Generates** the full kit in parallel — blog post, social threads, OG images, FAQ, product video
5. **Reviews** everything as a creative director — scores assets, requests revisions, ensures coherence
6. **Adapts** from your feedback — approvals, rejections, and edits feed future recommendations

It's not a prompt template. It's a multi-agent system with a real pipeline, real persistence, and a feedback-informed strategy loop.

---

## How It Works

```
Paste URL  →  Analyze  →  Research  →  Strategize  →  Generate (parallel)  →  Review  →  Done
                                                       ↓
                                                  Blog · Social · FAQ · Images · Video
```

The pipeline is event-driven via BullMQ. Each phase publishes progress to Redis pub/sub, which the dashboard streams to the user via Server-Sent Events. You watch the AI team work in real time.

## Architecture — and Why

Five Render services, each with one job:

| Service | Type | Role |
|---|---|---|
| `launchkit-web` | Web | Hono API + React SPA + webhook receiver + SSE |
| `launchkit-worker` | Worker | BullMQ processor + Anthropic agentic loops + fal.ai |
| `launchkit-cron` | Cron | Repo monitor + nightly learning aggregation + cleanup |
| `launchkit-redis` | Key-Value | BullMQ queue + pub/sub for SSE + GitHub API cache |
| `launchkit-db` | Postgres | Durable state + pgvector embeddings for similarity |

### Why five services?

**Web ↔ Worker separation** is non-negotiable. AI agentic loops take 30-180 seconds. They cannot run in HTTP request handlers. The worker exists so the web service stays snappy.

**Cron exists** because the self-learning system needs periodic aggregation, and webhook-enabled repos need a fallback poller for missed events.

**Redis serves three roles:**
1. BullMQ queue backing store
2. Pub/sub channel for real-time SSE updates from worker → web → browser
3. GitHub API response cache (rate limit protection)

Each role uses a different Redis primitive. Adding a separate cache or queue service would just be more things to break.

### Why Postgres AND Redis?

Different consistency needs. Redis is fast and ephemeral — perfect for queues and pub/sub. Postgres is durable and transactional — perfect for projects, assets, and embeddings. Trying to use one for both leads to either lost jobs (Postgres queue is slow and lossy) or lost data (Redis as primary store is fragile).

### Why pgvector instead of a dedicated vector DB?

The self-learning system finds similar past projects via cosine similarity on project embeddings. This is one query pattern. Adding Pinecone or Weaviate would mean another service, another credential, another failure mode. pgvector lets one Postgres instance do double duty with an HNSW index — fast enough for this scale, with zero operational overhead.

```sql
CREATE INDEX ON projects USING hnsw (embedding vector_cosine_ops);
```

### Why BullMQ instead of a simple queue?

The pipeline isn't linear. After strategizing, the worker fans out 5-8 generation jobs in parallel. Each has different priorities (video is expensive and slow), retries (text is cheap and idempotent), and timeouts. After all generation jobs complete, a single review job runs. That's a job dependency graph, not a queue.

BullMQ handles all of this — priorities, retries, timeouts, fan-out, completion detection — with a few lines of config.

### Why Claude tool use instead of an agent framework?

The research agent is the only truly agentic component. It uses Claude's native tool use to autonomously decide what to search and when to stop. Total implementation: ~50 lines (`apps/worker/src/agents/launch-research-agent.ts`).

Frameworks like LangChain would add 100MB of dependencies and hide the loop behind abstractions. Claude's SDK gives us the loop directly.

```typescript
for (let step = 0; step < maxSteps; step++) {
  const response = await anthropic.messages.create({ ... tools, messages });
  if (response.stop_reason === 'end_turn') return finalResult();
  // execute tool, append result, continue
}
```

Strategy and content agents are *prompted* (single Claude call with rich context) — they don't need a loop, just good prompts. Mixing agentic and prompted patterns where each fits is the right call.

## The Self-Learning System

This is the part that helps LaunchKit adapt recommendations over time.

```
User feedback (approve/reject/edit) → stored on assets table
                                              ↓
                              Cron job (every 6 hours)
                                              ↓
                              Aggregate by project category
                                              ↓
                  "CLI tools: technical tone +35% approval"
                  "Web apps: video drives 2x review scores"
                  "Users delete LinkedIn 60% of the time for libraries"
                                              ↓
                              Stored in strategy_insights table
                                              ↓
                Next project: strategist reads insights + pgvector finds
                similar past projects → makes better decisions
```

The strategist agent receives these insights as part of its prompt context. Over time, the system adapts which content types and tones it favors for different categories.

The seeded database includes example insights so you can see the system in action immediately.

## Deploy Your Own

### Prerequisites

- An [Anthropic API key](https://console.anthropic.com)
- A [fal.ai API key](https://fal.ai) (optional — text generation works without it)
- A Render account

### One-Click Deploy

1. Fork this repo
2. In Render, create a new Blueprint and connect your fork
3. Render reads `render.yaml` and provisions all five services
4. Enter your API keys when prompted
5. Wait ~3 minutes for everything to start

That's it. Your LaunchKit is live.

The Blueprint also runs database migrations before each web deploy, so a fresh Render setup comes up with the schema in place.

### Environment Variables

| Variable | Required | Purpose |
|---|---|---|
| `ANTHROPIC_API_KEY` | Yes | Claude API access for all agents |
| `ANTHROPIC_MODEL` | No | Override the Claude model used by the worker |
| `FAL_API_KEY` | No* | fal.ai for image and video generation |
| `GITHUB_WEBHOOK_SECRET` | No | Required only for webhook integration |
| `GITHUB_TOKEN` | Recommended | Raises GitHub API limits for analysis, research, and cron polling |
| `WEB_SEARCH_PROVIDER` | No | Research search backend. Defaults to `duckduckgo` |
| `API_KEY` | No | Optional bearer token required by the API when set |
| `DATABASE_URL` | Auto | Provided by Render Postgres |
| `REDIS_URL` | Auto | Provided by Render Key-Value |

*Without a fal.ai key, image/video assets are skipped and the kit ships text-only. Everything else works.

## Local Development

```bash
# Clone and install
git clone <your-fork>
cd launchkit
npm install

# Start Postgres + Redis locally
docker compose up -d

# Set up environment
cp .env.example .env
# Edit .env with your API keys

# Initialize the database
psql $DATABASE_URL < migrations/0001_init.sql

# Seed with demo data
npm run seed

# Run all services concurrently
npm run dev
```

Visit `http://localhost:5173` for the dashboard. The API is at `http://localhost:3000`.

```
    │   └── lib/             #   database, redis-client, job-queue-clients
    │
    ├── worker/              # BullMQ processor
    │   ├── processors/      #   analyze-project-repository, build-launch-strategy, generate-project-assets, ...
    │   ├── agents/          #   launch-research-agent, launch-strategy-agent, written-asset-agent, ...
    │   ├── tools/           #   github-repository-tools, web-research-tools, project-insight-memory
    │   └── lib/             #   anthropic-claude-client, fal-media-client, project-embedding-service, ...
    │
    ├── cron/                # Periodic tasks
    │   ├── sync-github-project-activity.ts   #   webhook fallback poller
    │   ├── aggregate-feedback-insights.ts    #   strategy insight aggregation
    │   └── cleanup-stale-launch-data.ts      #   stale data cleanup
    │
    └── dashboard/           # React SPA
        ├── components/      #   ProjectSummaryList, ProjectDetailView, GeneratedAssetCard, ...
        ├── hooks/           #   useProjectData, useProjectEventStream

### Naming Conventions

- `ProjectProgressPublisher` means internal Redis/SSE progress events.
- `SocialPublisher` means external posting to a platform such as X or LinkedIn.
- `generationInstructions` means the exact instructions used to create or regenerate an asset.
- Role-based agent names describe the AI persona; processor names describe the concrete system action they perform.
        └── lib/             #   API client
```
